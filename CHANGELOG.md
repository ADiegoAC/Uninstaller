# Changelog — Un1nst4ll3r

> Cada grupo (**Motor**, **UI**, **Integração**) evolui com seu próprio versionamento independente. O número do título de cada bloco representa a timeline combinada do projeto; a tag `[Grupo - x.y.z]` indica a versão real daquele componente.

**Versões atuais:** Motor `3.1.1` · UI `1.5.1` · Integração `2.0.0`

---

## [0.1.0] O Início
- **[Motor - 0.1.0]** Arquitetura base de retorno via `PSCustomObject`. Scan básico do Registro do Windows (Win32) e varredura nas 4 chaves padrão (`HKLM/HKCU` e `WOW6432Node`).
- **[Motor - 0.1.1]** Filtro inicial de componentes do sistema (`SystemComponent`) e chaves pai (`ParentKeyName`).
- **[UI - 0.1.0]** Esboço inicial da janela principal e DataGridView.
- **[Integração - 0.1.0]** Primeiro Dot-Sourcing (`. .\Un1nst4ll3r.ps1`) chamando a função de scan e preenchendo o Grid de forma bruta.

## [0.2.0] Microsoft Store (AppX)
- **[Motor - 0.2.0]** Integração com a Microsoft Store (AppX) via `Get-AppxPackage`. Filtro de Frameworks base (`IsFramework -eq $false`) e apps não assinados (`SignatureKind -ne "None"`).

## [0.3.0] Filtros de System Apps
- **[Motor - 0.2.1]** Filtro de Aplicativos Invisíveis do Sistema (Oculta apps da Microsoft onde `AppListEntry = none`). Filtro de Componentes Nativos (oculta apps cujo fabricante é exatamente "Microsoft Windows") e filtro de GUIDs puros (ex: `1527c705-...`) para remover ruído no Grid.

## [0.4.0] Fallback Fabricante e Design Base
- **[Motor - 0.2.2]** Fallback de Fabricante: leitura de metadados do arquivo `.exe` via `[System.Diagnostics.FileVersionInfo]` quando o registro não fornece o `Publisher`. Limpeza de strings de fabricantes de apps da Store (remoção do prefixo `CN=`).
- **[UI - 0.2.0]** Criação do design base: Painel de cabeçalho, barra de ações e rodapé (Status Bar). Truque de DPI Awareness (`SetProcessDPIAware`) via C# embutido para evitar embaçamento em monitores com zoom (125%, 150%).

## [0.5.0] Deep Size e Banco MSI
- **[Motor - 0.3.0]** Função `Get-Un1nst4ll3rDeepSize` (Fase 2). Cálculo de tamanho físico em disco via `Get-ChildItem -Recurse -Force`, executado apenas quando o `EstimatedSize` do registro é inexistente. Variável de ambiente `Local` adicionada ao retorno do Motor para suportar cálculos e checagens futuras (Deep Size e Órfãos).
- **[Motor - 0.3.1]** Consulta ao Banco de Dados MSI (Tentativa 2.5) via COM Object `WindowsInstaller.Installer` para ler `InstallLocation` e `InstallSource`. Filtros Anti-Lixo MSI (Package Cache, `Windows\TEMP`, pastas IXP e pastas hexadecimais raiz, ex: `c:\61bec893cfb98cb796e5\`). Roubo de Palavra-chave do Cache: se o `UninstallString` aponta para o Package Cache, extrai a palavra inicial (ex: "python" de `python-3.14.5-amd64.exe`) e usa essa palavra como prioridade na busca em disco, recusando o Cache como local de instalação. Adição de variáveis de ambiente nativas e caminhos críticos: `$env:LOCALAPPDATA\Programs` (padrão para Python, VSCode etc.), `$env:ProgramFiles`, `$env:SystemDrive` e `$env:SystemRoot` (para suportar instalações em drives diferentes de `C:\`).

## [0.6.0] Metadados, Exportação e Ordenação
- **[Motor - 0.3.2]** Metadados Expandidos: Propriedades `InstallDate`, `HelpLink`, `UninstallString`, `NoRemove`, `NoModify` e `NoRepair` extraídas do Registro e retornadas no objeto.
- **[Integração - 0.2.0]** Exportação de JSON: A UI gera automaticamente o arquivo `Un1nst4ll3r_ScanResult.json` na mesma pasta do script após o scan, permitindo análise forense externa e debug de heurística.
- **[UI - 0.2.1]** Auto-Resize da coluna Local com `AutoSizeMode = Fill` e `MinimumWidth = 150`, garantindo que ela ocupe o espaço restante sem criar barra de rolagem horizontal. Controle Manual de Ordenação via variáveis de estado (`$script:lastSortedColumn`, `$script:lastSortDirection`) para grids não vinculados a bancos de dados — o WinForms perde o estado de ordenação nesses casos, e o fix garante que cliques consecutivos no header invertam a ordem corretamente.

## [0.7.0] Heurística, Atalhos e Órfãos
- **[Motor - 0.4.0]** Heurística de Caminho (Fases 3 e 4) para apps sem `InstallLocation`. Parser avançado do `UninstallString` (Regex para parâmetros `-f"...\.isu"` do InstallShield antigo). Parser de Atalhos (`.lnk`) do Menu Iniciar e Desktop via `WScript.Shell`. Detecção de programas órfãos (pasta inexistente no disco).
- **[Motor - 0.4.1]** Bug de codificação (Encoding UTF-8 vs Windows-1252): alteração de "Órfão" para "Orfao" para evitar quebras de parser (Token inesperado) em sistemas Windows em português.
- **[Motor - 0.4.2]** `Get-NormalizedName`: normalização de strings removendo espaços, traços e underscores e convertendo para minúsculas, permitindo que "Logicmaster 90-30" corresponda a "LM90.exe". Detecção de Runtimes e Drivers via regex (`Redistributable|Runtime|Framework|Driver|...`), que passam a receber `Status = "OK"` e `ExePath = ""`, eliminando falsos positivos. Busca em Pastas Pai: se `InstallLocation` aponta para subpasta genérica (ex: `\Support`, `\Bin`), inclui a pasta pai na busca do EXE; se o EXE é encontrado lá, `$prog.Local` é atualizado para a pasta raiz, garantindo cálculo de tamanho completo. Nova Lógica de Status: `OK` (pasta existe e possui EXE, ou é Runtime detectado), `Vestigio` (pasta existe mas sem EXE válido — apenas DLLs/configs), `Orfao` (pasta registrada não existe no disco) e `SemLocal` (registro não forneceu `InstallLocation` e a heurística falhou).
- **[Motor - 0.4.3]** Função `Find-Un1nst4ll3rMainExe` (Fallback Inteligente): busca sequencial na raiz e na primeira camada de subpastas, priorizando diretórios `bin` ou com o nome do app. Extração de palavra-chave para busca de EXEs (ex: "VLC media player" busca por `*VLC*.exe`). Fallback de Menor Nome: se nenhum nome de EXE bater com o app, escolhe o `.exe` válido com o menor nome no diretório (ex: `vlc.exe` vence `vlc-cache-gen.exe`). Propriedade `ExePath` adicionada ao retorno do Motor. Blacklist de EXEs (`uninstall`, `unins`, `setup`, `helper`, `crash`) na busca do executável principal. Refinamento do Deep Size: o cálculo físico em disco passa a ser ignorado quando o Registro já forneceu `EstimatedSize`, economizando tempo e evitando medições incorretas em pastas compartilhadas.
- **[UI - 0.2.2]** Coloração de linhas de programas órfãos em vermelho no Grid.

## [1.4.0] Separação Radical de Escopos
- **[Motor - 2.0.0]** Separação Radical de Escopos (Fase 1 vs Fase 2: Local vs ExePath): reestruturação completa do fluxo de execução — a Fase 1 consolida e fixa unicamente o diretório físico (`Local`); só então a Fase 2 inicia a determinação do executável principal (`ExePath`), eliminando o conflito de variáveis e os loops cegos herdados da v2.2. Heurística de Persistência (Não-Desistência) em `Find-Un1nst4ll3rMainExe`: ao entrar em subpasta inválida (ex: `\uninstall` contendo apenas arquivos da blacklist), ignora o bloco morto e continua a busca em subpastas restantes (`\bin`, `\App`) até esgotar as possibilidades físicas do disco. Navegação Pai Inteligente: se o caminho extraído do registro aponta para subpastas genéricas (ex: `\Support`, `\bin`), o motor sobe um nível (`..`) para varrer a pasta raiz do aplicativo, resolvendo casos crônicos como o do Logicmaster 90-30. Blacklist Dinâmica Baseada no `UninstallString`: limpa aspas e parâmetros do desinstalador oficial registrado para banir estritamente aquele arquivo específico (ex: `unins000.exe`), sem bloquear executáveis legítimos que carregam "Uninstaller" no próprio nome (ex: Smarty Uninstaller 4). Mapeamento de Inicializadores Não-Executáveis (`.bat`/`.cmd`) na fase de desempate, evitando classificação incorreta como "Vestígio". Refinamento do campo "Iniciar Em" de Atalhos: o parser `.lnk` passa a ler e priorizar `WorkingDirectory` antes de `TargetPath`, isolando a pasta real de instalação de softwares que usam wrappers/redirecionadores em `System32`. Substituição de variáveis de ambiente por métodos .NET (`[System.Environment]::GetFolderPath()`), blindando o script contra desvios e redirecionamentos 32-bit vs 64-bit (`SysWOW64`). (Breaking Change arquitetural)
- **[Motor - 2.0.1]** Tratamento de Exceções e Ordenação de Runtimes (Força `Status = "OK"`, `ExePath = ""` e `Tamanho = 0`). Bug de Sintaxe no Match de Runtimes: ponto de propriedade ausente no operador de correspondência, `$Prog Nome` corrigido para `$Prog.Nome`. Limpeza de Escopo via PSScriptAnalyzer: remoção das variáveis órfãs e redundantes `localFound` e `calcSize`.
- **[Integração - 1.4.0]** Motor de Depuração Forense (JSON Live no Terminal): interceptador dinâmico de logs que imprime no terminal um bloco JSON estruturado para cada `DisplayIcon`/atalho analisado, detalhando as propriedades extraídas (`Destino`, `IniciarEm`, `FoiUsado`) e o motivo exato de um eventual descarte. - **[Integração - 1.5.1]** Fluxo de dados validado com a propriedade `ExePath`, garantindo que a UI receba o caminho do executável principal sem depender de ícones do Windows.

Sistema de Evidências Correlacionadas para consolidação e validação cruzada entre fontes. Classificação de Evidências por Confiabilidade: pesos internos para priorização automática das fontes quando há múltiplos caminhos candidatos. (Breaking Change conceitual)
- **[Motor - 3.0.1]** Modularização Completa da Cadeia de Descoberta: separação em motores especializados independentes (Registry, MSI, Shortcut, DisplayIcon, Process e Deep Discovery), reduzindo acoplamentos. Validação Física Obrigatória de Caminhos: o motor deixa de confiar exclusivamente no Registro e valida fisicamente todas as evidências antes da localização final. Refinamento da Descoberta de Diretórios de Instalação (correlação entre `InstallLocation`, `InstallSource`, `DisplayIcon`, atalhos e executáveis ativos para softwares legados/corporativos). Otimização de Performance e Reutilização de Evidências, eliminando consultas redundantes ao disco, Registro e MSI.

Compatibilidade Aprimorada com Aplicações Corporativas instaladas via MSI, InstallShield, wrappers proprietários e distribuidores corporativos com metadados inconsistentes.

Normalização de Dados Multi-Fonte, eliminando duplicidades e conflitos entre Registro, MSI, atalhos, processos e artefatos do sistema. Ampliação da Base de Inventário Persistida: os campos `Publisher`, `ProductCode`, `UpgradeCode`, `InstallSource`, `InstallLocation`, `EstimatedSize`, `QuietUninstallString` e `DisplayIcon` passam a integrar a estrutura persistida utilizada pela UI.

Ampliação da Observabilidade Operacional: integração expandida com o framework de logs categorizados. ## [2.2.0] Correção Crítica MuiCache e Deduplicação
- **[Motor - 3.1.1]** Bug Crítico de Desempacotamento do MuiCache (Object Unpacking): após `Get-Un1nst4ll3rMuiCache` passar a retornar objetos estruturados (`@{ExePath=...; RegKeyName=...}`) em vez de strings, `Find-Un1nst4ll3rOrphans` e `Get-Un1nst4ll3rDeepSize` falhavam silenciosamente ao passar o objeto para `Test-Path`/`Split-Path`, descartando apps válidos (ex: DOSBox) como "ExePath missing" ou logando o objeto inteiro. Implementada rotina de desempacotamento (`is [PSCustomObject]`) para extrair a string do caminho antes da validação. Parâmetro Inexistente na Chamada do Orphan Scanner: `Get-Un1nst4ll3rScan` invocava `Find-Un1nst4ll3rOrphans` com o parâmetro incorreto `-KnownApps`, corrigido para `-ResolvedPrograms` (conforme assinatura da função), o que quebrava a deduplicação hierárquica. Eliminação dos "Buracos Negros" de Log: adicionados logs (`Write-Un1Log`) em todos os pontos de saída silenciosa (`continue`) dentro de `Find-Un1nst4ll3rOrphans`, categorizados como `[ORPHAN]` em DarkGray. Precisão Semântica em Logs de Deduplicação: mensagem alterada de "Skipped (Name already in Registry)" para "Skipped (Name already mapped in current scan)", refletindo que a lista de entrada inclui apps de outras fontes além do Registro.
- **[UI - 1.6.1]** Deduplicação Duplicada no Phase 2.5: removido o bloco "Phase 2.5" da função `Update-Grid` na Interface, que executava `Find-Un1nst4ll3rOrphans` uma segunda vez. Como o scan de órfãos já está integrado ao Motor (Bloco 1.1), a execução redundante na UI fazia órfãos válidos, já mapeados pelo Motor, serem detectados como duplicatas e logados incorretamente.

## [2.3.0] Pipeline de Ícones e Resolução de Apps Modernos (Electron)
- **[Motor - 3.2.0]** Atualização da Heurística de Updaters (Updater Golden Rule): Aplicações baseadas em Squirrel.js/Electron (Discord, Slack, VS Code, Teams) utilizam `Update.exe` como alvo nos atalhos e registro. O motor agora detecta padrões de updaters (`Update.exe`, `Updater.exe`, etc.) e os rejeita como `ExePath` principal, forçando a heurística interna (`Find-Un1nst4ll3rMainExe`) a encontrar o executável real dentro das subpastas versionadas (ex: `app-1.0.9240\Discord.exe`). O updater é colocado na blacklist da busca, garantindo que o `ExePath` final seja o processo principal do aplicativo. Desacoplamento de Evidências de Atalhos (PRIORIDADE 3): A coleta de `IconLocation`, `ShortcutTitle` e metadados de atalhos agora ocorre imediatamente ao encontrar um match, independentemente da validade do `TargetPath`. Isso impede que aplicativos com alvos inválidos ou updaters percam a referência do seu ícone real. Fallback de Diretório via `IconLocation`: Se o `TargetPath` do atalho falhar, mas o `IconLocation` apontar para um arquivo válido no disco (ex: `app.ico`), o diretório desse ícone é promovido a candidato de instalação (`guessedPath`). Expansão do Cache de Atalhos (`Get-Un1nst4ll3rShortcutCache`): Captura agora a propriedade `Arguments` dos atalhos (crucial para mapear argumentos como `--processStart`). Inclusão da pasta de atalhos fixados na Barra de Tarefas (`TaskBar`) no escopo de varredura. Expansão automática de variáveis de ambiente (`%LocalAppData%`, `%USERPROFILE%`) nos caminhos de `TargetPath`, `WorkingDirectory` e `IconLocation` no momento da construção do cache, eliminando falhas de resolução na UI. Inclusão de atalhos sem `TargetPath` válido mas com `IconLocation` definido (comum em apps da Store/protocolos web).
- **[UI - 1.7.0]** Reestruturação da Pipeline de Extração de Ícones (`Get-Un1nst4ll3rAppIcon`): Mudança de paradigma — a UI deixa de "adivinhar" ícones via heurísticas pesadas e passa a consumir preferencialmente as evidências pré-resolvidas pelo Motor. Reordenação de Estratégias baseada em Confiança: S1 (Registro) -> S2 (Ícone explícito do atalho via `ShortcutIconLocations`) -> S3 (SysPkgBank) -> S4 (Windows Installer Cache) -> S5 (Heurística de regex com cache global). Suporte a `ShortcutIconLocations` (S2): A UI agora recebe o array de ícones extraídos dos atalhos pelo Motor, priorizando a extração a partir deles antes de qualquer busca em disco. Resolução do Windows Installer Cache (S4): Implementada extração de ícones diretamente da pasta oculta `C:\Windows\Installer\{GUID}\`, utilizando a propriedade `Chave` do registro. Inclui lógica para leitura de arquivos de ícone sem extensão através do construtor raw `System.Drawing.Icon`, contornando limitações do `ExtractAssociatedIcon`. Suporte a `Arguments` no Cache de Atalhos: A interface de extração de ícones agora utiliza os argumentos dos atalhos (quando disponíveis) como fallback na identificação de executáveis principais ocultos por Updaters. Descarte automático de sufixos de índice de ícone (ex: `,0`, `,-1`) durante o parse de caminhos de ícones. 
- **[Integração - 2.3.0]** Expansão do Schema do Cache JSON: Inclusão e população confiável da propriedade `ShortcutIconLocations` (array) e `Arguments` nos registros persistidos. Passagem explícita de parâmetros forenses (`Chave`, `ShortcutIconLocations`) da camada de dados (`Load-GridFromJson`) para a camada de renderização de ícones (`Get-Un1nst4ll3rAppIcon`), fechando a lacuna de dados que causava ícones faltantes ou incorretos em aplicações modernas. Consolidação do fluxo onde o Motor é a fonte única de verdade para metadados, e a UI atua meramente como renderizadora stateless dessas evidências.
